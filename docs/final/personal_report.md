## 3. 关键实现内容

### 3.1 串行搜索内核

串行部分从 TSPLIB95 解析器开始。程序需要读取 `.tsp` 文件中的实例名称、城市数、边权类型和坐标数据，并转换为统一的距离矩阵。距离矩阵采用一维连续数组存储，这样既便于 CPU 端缓存访问，也便于 CUDA 后端复制到设备内存。路径使用城市编号排列表示，并配合合法性检查，确保 2-opt 操作后仍是一个完整排列。

SA 内核的关键优化是 2-opt 增量计算。一次 2-opt 移动只改变四条边，因此路径长度变化可以用四次距离访问得到，而不需要重新遍历整条路径。该优化把一次候选评价从 $O(n)$ 降到 $O(1)$。当实验包含几十条搜索链和百万级迭代时，这个细节直接影响程序能否在合理时间内完成。

在 QLSA 部分，我实现了状态、动作、奖励和 Q 表更新，并支持 epsilon-greedy 与 softmax 两类动作选择策略。默认工程变体之外，还实现了 `paper` 和 `paper-sb` 两个机制对齐入口：`paper` 变体加入 current、global best、random、double-bridge 四类候选来源；`paper-sb` 在此基础上加入基于路径差异的 diversity state。这样可以把参考论文中的 candidate-leader 和 State-Based QLSA 思想纳入同一套 C++ 实验框架。

### 3.2 OpenMP 多链并行

OpenMP 是本项目最主要的性能后端。我采用链级任务划分：每个 `chain_id` 对应一条独立搜索链，线程按照静态调度领取链编号并运行本地搜索。链内路径、随机数、计数器和 Q 表均为私有数据，距离矩阵为只读共享数据，所有链完成后再归约全局最优结果。

这种设计避免了在 2-opt 内层循环中频繁同步。一次 2-opt 候选评价很轻，如果把单次移动拆给多个 CPU 线程，线程调度和同步开销很容易超过计算本身；把完整搜索链作为任务后，同步次数从“迭代次数级别”下降到“搜索链数量级别”。这也是 OpenMP 实验能够取得稳定加速的主要原因。

### 3.3 CUDA 后端

CUDA 后端分为多链模式和候选批量评价模式。多链模式保持“一条 GPU 线程块对应一条搜索链”的结构，用于验证 GPU 可以并发执行多条独立搜索链。候选批量评价模式进一步让线程块内多个线程同时生成和评价 2-opt 候选，并通过共享内存和块内归约选择移动。

为了减少路径操作开销，我实现了并行路径反转、最优路径协作复制，以及 `best`、`random`、`hybrid` 三种候选选择策略。QLSA 也接入了 CUDA candidate 路径。实验结果显示，candidate-best 在 ch130、a280、lin318、rat575 等实例上改善了路径质量，但运行时间仍受到路径反转、同步和全局内存访问影响。因此，CUDA 在本项目中主要体现 GPU 端搜索增强和工程扩展价值。

### 3.4 MPI + OpenMP 分布式并行

MPI + OpenMP 后端用于验证分布式内存环境下的链级并行。两台 Ubuntu 虚拟机中，每个 MPI 进程负责一段连续搜索链，进程内部继续使用 OpenMP 运行本地链。进程之间不交换每一步搜索状态，只在结束阶段汇集本地最优路径和统计量，再广播全局最优路径。

这部分工作涉及虚拟机网络、SSH、Open MPI、hostfile、远端构建和结果同步。最终实验通过真实 `mpirun` 完成，能够区分 np=1、np=2、本地 OpenMP 与 MPI 执行路径。该实验让我更清楚地理解了粗粒度任务划分在分布式环境中的意义：只要搜索过程中不频繁通信，MPI 进程之间的通信量就可以控制在较低水平。

## 4. 实验与分析工作

实验按问题分层设计。默认参数多实例实验用于评价 OpenMP 加速效果；调优验证和定向增强用于观察困难实例的解质量变化；策略对比和机制对齐实验用于分析 QLSA 设计因素；CUDA 实验用于比较多链模式、候选批量评价、候选策略和路径反转；MPI 实验用于检查双虚拟机分布式运行；大实例压力测试用于验证工程在更大 TSPLIB95 实例上的运行能力。

默认参数实验给出了最稳定的性能结论。SA 在 6 个 TSPLIB95 实例上的平均加速比为 5.46，平均并行效率为 68.28%；QLSA 的平均加速比为 4.98，平均并行效率为 62.29%。这些结果说明链级 OpenMP 并行能够有效利用多核 CPU，同时保持与串行多链相同的搜索预算。

解质量分析中，我把默认参数结果和高预算结果分开处理。默认参数下 berlin52、eil51、st70 能达到 BKS，而 eil76、rat99、eil101 仍存在偏差。经过调优验证和定向增强后，eil101 中 SA 和 QLSA 均达到 BKS=629；rat99 中 QLSA high-budget 达到 BKS=1211，而 SA high-budget 最好为 1212。这部分结果说明 QLSA 在部分困难实例上能够带来质量收益，同时也说明参数和搜索预算对结果有明显影响。

CUDA 实验的分析重点是质量与时间的权衡。candidate-best 会在一批候选中选择增量更优的移动，因此在若干实例上降低了 Gap；candidate-random 更接近随机提案过程，时间更低但质量收益较弱；hybrid 用于观察二者之间的折中。Nsight Systems 记录显示，当前配置下主机与设备之间的数据传输规模很小，主要等待来自设备端 kernel 执行。这帮助我把 CUDA 结果解释为候选批量评价和路径操作成本之间的权衡，而不是简单地比较后端快慢。

MPI 实验在两台 Ubuntu 虚拟机中完成真实 `mpirun` 运行。berlin52 formal scaling 中，SA 的 np=2 接近 2 倍加速，QLSA 的 np=2 加速比约为 1.70 至 1.82，通信开销为毫秒级。该实验验证了 MPI 进程级链分配、进程内 OpenMP 并行和最终归约能够共同工作。

大实例压力测试覆盖了 38 个可用 TSPLIB95 实例，并在课程报告中选取 10 个代表实例分析。对于 dsj1000、u1060、vm1084 等千点级实例，实验使用百万迭代、64 条搜索链、8 线程和 3 次重复运行。结果显示，千点级实例在相同预算下仍有较高偏差，但 QLSA 在这三个代表实例上的最小偏差均低于 SA。这部分实验主要用于说明工程可扩展性和规模变化趋势。

## 5. 遇到的问题和解决方法

第一个问题是 CUDA 中计算密度和路径操作之间的矛盾。候选评价适合并行，但路径反转会修改同一条路径数组。实现时我先采用安全的串行反转，再加入块内并行反转和最优路径协作复制，并通过测试保证路径合法性。实验结果显示，候选批量评价可以改善部分实例的解质量，但路径操作和同步仍然是主要代价。

第二个问题是 MPI 双虚拟机实验的可信度。为了证明结果来自真实分布式运行，需要配置 SSH、Open MPI 前缀、hostfile、远端项目同步和 VM 端构建。运行过程中还要避免把 fallback 或单机结果写成 MPI 结果。最终通过 hostname 测试、`mpirun` 输出、CSV 结果和通信时间记录确认了实验来源。

## 6. 收获与不足

这次大作业让我更具体地理解了并行算法中的任务划分问题。并行化并不是把所有循环都拆开，而是要根据计算粒度、同步代价和数据共享方式选择合适层次。SA/QLSA 的搜索链天然独立，因此链级并行适合 OpenMP 和 MPI；CUDA 需要更多线程参与，才进一步引入块内候选批量评价。不同后端的设计目标不同，不能只用同一项指标评价。

第二个收获是对实验结论边界的认识更清楚。OpenMP 是本项目最稳定的性能后端；QLSA 在部分困难实例和较高预算下改善解质量；CUDA 展示了 GPU 端候选批量评价的工程可能；MPI 说明多链搜索可以扩展到分布式内存环境。每个结论都有对应数据，也都有适用范围。

第三个收获是可复现实验流程的重要性。项目中关键实验都保存原始结果和汇总结果，图表从 CSV 生成，报告检查脚本负责检查图片路径、编码和敏感信息。这个流程在后期反复修改报告和筛选图表时很有价值，也减少了手工复制数据造成的错误。

项目仍有不足。`paper` / `paper-sb` 机制对齐实验已经完成代表实例验证，但还没有像默认变体那样进行系统参数搜索；CUDA 后端虽然加入了候选策略、并行反转和 profiling 入口，但更适合 GPU 的路径布局仍有继续优化空间；MPI 实验只在两台虚拟机中完成，规模和网络环境有限。后续如果继续推进，可以重点研究分布式搜索中的周期性最优路径交换，以及更多重复次数下的统计检验。
