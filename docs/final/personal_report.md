## 2. 个人承担的工作

本次大作业为单人项目，因此从选题、论文阅读、工程实现到实验分析和报告整理均由本人完成。选题阶段主要考虑两个因素：一是课程要求强调并行算法与实际应用或近期论文的结合；二是旅行推销员问题有标准测试集和已知最优值，便于比较算法质量。最终选择以 Q-Learning 辅助模拟退火求解 TSP 的论文为参考，在 C++ 工程中实现 SA、QLSA 以及多种并行后端。

项目推进过程中，本人承担了完整的代码设计和实验设计工作。前期重点是搭建可维护的串行基线，包括 TSPLIB95 数据解析、距离矩阵、路径表示、2-opt 增量计算和模拟退火主循环。随后在串行基线之上实现 QLSA，并增加 epsilon-greedy 和 softmax 两种动作选择策略。并行化阶段先完成 OpenMP 多搜索链并行，再扩展 CUDA 和 MPI + OpenMP。实验阶段编写了多组自动运行和分析脚本，用于生成原始结果表、汇总结果表、图表和报告素材。

## 3. 关键实现内容

在串行部分，主要工作是把 TSP 搜索过程拆成清晰的底层模块。距离矩阵采用一维连续数组存储，便于 CPU 缓存访问和 GPU 数据复制；路径表示使用城市编号排列，并提供合法性检查；2-opt 操作使用四条边的距离差计算增量，避免每次扰动后完整重算路径长度。这些实现看起来是基础工作，但如果没有这些优化，后续多链并行会被单链开销拖慢。

在 QLSA 部分，本人实现了 Q 表、状态离散、动作集合、奖励计算和策略选择。实现时没有把 C++ 主线写成参考论文中 State-Based QLSA 的逐项复刻，而是按照课程项目的工程目标，将论文中的学习辅助搜索思想迁移到可运行、可测试、可并行的 C++ 结构中。该处理方式在报告中也进行了说明，避免把实现范围夸大。

OpenMP 是本项目最重要的并行后端。实现时选择以搜索链为并行粒度，每条链维护独立路径、随机数状态和最优结果，距离矩阵只读共享。这样的设计减少了锁和同步开销，也便于保持结果可复现。CUDA 部分实现了多链模式、候选 2-opt 批量评价和并行路径反转；MPI 部分实现了双虚拟机环境下的 MPI + OpenMP 分布式多链运行。CUDA 和 MPI 的结果没有被写成主性能结论，而是作为工程扩展和并行体系探索。

## 4. 实验与分析工作

实验工作分为默认参数实验、调参实验、定向增强实验、策略对比、CUDA 实验、MPI 实验和大实例压力测试。默认参数实验用于评价 OpenMP 多链并行的加速效果；调参和定向增强实验用于观察困难实例上的解质量变化；策略对比用于比较 QLSA 中 epsilon-greedy 和 softmax 的效果；CUDA 实验用于分析 GPU 后端的质量和运行时间边界；MPI 实验用于验证分布式内存环境下的链级划分方式。

实验分析中尽量把结论与数据对应起来。例如，OpenMP 默认实验得到 SA 平均加速比 5.46、QLSA 平均加速比 4.98，因此报告将 OpenMP 作为主要性能结论。rat99 的定向增强实验中，QLSA 达到 BKS=1211，而 SA 最好为 1212，因此报告将其作为 QLSA 在部分实例上改善解质量的例子，但没有写成 QLSA 在所有情况下都优于 SA。CUDA 候选批量评价在部分实例上降低了最小偏差，但运行时间仍高于 OpenMP，因此报告只把 CUDA 写作工程扩展和后续优化方向。

除实验运行外，本人还负责结果整理和图表生成。项目中每组重要实验都保存原始结果表和汇总结果表，图表从汇总结果表生成。报告后期对图片进行了筛选，删除或弃用不适合正文的架构示意图，保留能够直接支撑结论的实验图，避免报告变成文件清单或脚本清单。

## 5. 遇到的问题和解决方法

项目实施中遇到的第一个问题是数据准备。TSPLIB95 的部分镜像源不可用，下载脚本可能得到 HTML 页面而不是 `.tsp` 文件。为避免错误数据进入实验，后续增加了文件头检查、维度检查和 SHA256 记录，并在缺失数据时跳过实例而不是使脚本崩溃。

第二个问题是 CUDA 构建环境。Windows 下 Visual Studio 生成器识别 CUDA 工具链不稳定，最终改用 CMake + Ninja 构建 CUDA 版本，使 `cuda_kernels.cu` 能够真实编译。CUDA 后端虽然完成了多链、候选批量评价和并行路径反转，但实验表明小规模实例上仍不适合作为主性能结论。这个结果也让我认识到，GPU 并行并不是只要写出 kernel 就一定更快，数据规模、内存访问和同步方式都会影响结果。

第三个问题是 MPI 双虚拟机环境。为了让 MPI 真实跨虚拟机运行，需要配置 SSH、Open MPI 前缀、项目同步和 hostfile。运行过程中还必须区分本地 OpenMP 结果、单虚拟机 MPI 结果和双虚拟机 MPI 结果，不能把 fallback 或单机结果写成分布式实验。最终完成了真实 `mpirun` 运行，并记录通信时间和加速比。

第四个问题是报告整理。项目中后期生成了较多中间文档、旧版报告、图片和表格，容易让报告看起来像工程清单。最终整理时重新筛选核心结果，把主线收敛为“多搜索链并行化”，并且把个人工作说明从课程主报告中拆出，作为独立个人报告。

## 6. 收获与不足

这次项目的主要收获是理解了启发式搜索并行化中的粒度选择。相比把一次 2-opt 操作拆得很细，多搜索链并行更容易控制同步开销，也更适合在 OpenMP、CUDA 和 MPI 之间迁移。实验结果也说明，工程上可靠的并行方案不一定是最复杂的方案，OpenMP 多链并行反而是本项目中最稳定、最能支撑课程性能结论的部分。

另一个收获是对实验边界有了更清晰的认识。QLSA 在 rat99 上取得了比 SA 更好的 high-budget 结果，但不同实例上的表现仍受参数和预算影响；CUDA 实现了更多 GPU 端并行结构，但当前数据更适合说明工程扩展和质量探索；MPI 在双虚拟机上运行成功，体现的是分布式执行路径，而不是生产集群评测。报告写作必须把这些口径区分清楚，否则结果容易被过度解释。

项目仍有不足。首先，虽然已经补充 `paper` / `paper-sb` 代表实例实验，但它们还没有像默认变体那样完成大范围参数搜索，因此报告仍把历史主实验和机制对齐实验分开讨论。其次，CUDA 后端已经实现候选批量评价、候选策略和并行路径反转，但运行时间仍受路径操作、同步和全局内存访问影响，在更大规模实例上还需要继续优化。最后，MPI 部分只在两台虚拟机上验证，规模和网络环境都有限，后续可在更接近实际集群的环境中测试周期性交换最优路径等机制。
